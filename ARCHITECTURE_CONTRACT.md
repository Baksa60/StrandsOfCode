# ARCHITECTURE_CONTRACT.md

## 1. Цель

Обеспечить единый и однозначный контракт для конвертера кода StrandsOfCode, чтобы перенос, интеграция и рефакторинг:

- не ломали существующие форматы конвертации
- позволяли добавлять новые форматы и новые типы конвертеров
- не приводили к ложной унификации
- не размазывали ответственность между слоями
- гарантировали уменьшение технического долга, а не его увеличение

Создать архитектуру, в которой:

- существует один владелец конвертации (ConversionEngine)
- форматы не управляют состоянием, файлами и жизненным циклом
- добавление нового формата или нового типа конвертера не требует переписывания ядра
- перенос формата означает удаление старой логики, а не её оборачивание
- каждое изменение приводит к уменьшению количества строк кода и устранению дублирования

⚠️ **Контракт требует полной унификации интерфейса, за исключением нижеперечисленного.**

- Иконки форматов в UI
- Названия форматов в выпадающих списках
- Специфичные настройки формата в диалогах
- Текст в статусах ошибок конвертации

Этот контракт является обязательным для всех изменений в проекте.

## 2. Архитектурные слои и их власть

*(Остаются те же разделы 2.1–2.8, но с ужесточениями, перечисленными ниже)*

### 2.1. ConversionEngine — ЯДРО

**Дополнительные требования:**

- Engine обязан предоставлять API для подписки на прогресс, отмены, сохранения истории
- Любая логика, связанная с конвертацией, должна быть либо в Engine, либо в формате (как декларативное описание). Никакой промежуточной логики в UI или контроллерах

### 2.6. UI / Presentation — ФОРМАТ СЛЕП, UI — СПЕЦИАЛИЗИРОВАННЫЙ

**Дополнительные требования:**

- UI не может содержать никакой бизнес-логики (сбор файлов, конвертация, обработка результатов). Всё, что не относится к отображению и вводу, передаётся в Engine
- Запрещено создавать *Worker классы внутри UI. Все потоки должны управляться Engine

### 2.7. MainWindow — ЕДИНЫЙ ИНТЕРФЕЙС

**Дополнительные требования:**

- MainWindow не может содержать методов, превышающих 30 строк кода. Любая сложная логика должна быть вынесена в отдельные классы-менеджеры (например, SettingsManager, HistoryManager, FileSelectionManager)
- MainWindow не может импортировать конкретные форматы или конвертеры. Вся работа с форматами идёт через Engine и фабрики
- Запрещено хранить в MainWindow какие-либо данные о состоянии конвертации (списки файлов, опции и т.п.), кроме временного состояния для отображения. Всё состояние должно быть в Engine

### 2.8. Error Handling — ЕДИНЫЙ СТАНДАРТ

**Дополнительные требования:**

- Все ошибки, возникающие в форматах, должны возвращаться как ConversionEffect с типом error. Engine отвечает за их логирование, отображение и сохранение в историю
- Запрещено использовать try/except в UI для обработки ошибок конвертации — только через механизм Engine

## 3. Прямо запрещено

Агент НЕ ИМЕЕТ ПРАВА:

- создавать use<Format>Session
- создавать orchestrator'ы поверх ConversionEngine
- дублировать логику ConversionEngine
- делать «универсальный конвертер для всех форматов»
- сохранять старые API «на всякий случай»
- оборачивать старый код без удаления оригинала
- добавлять абстракции, не забирающие ответственность
- хардкодить сообщения об ошибках (исключая системные)
- оставлять закомментированный код
- создавать файлы с суффиксом _backup, _old, _deprecated в основном коде
- дублировать классы в разных файлах (например, ConversionWorker одновременно в main_window.py и conversion_worker.py)
- увеличивать общее количество строк кода в результате рефакторинга (допускается только уменьшение или сохранение при условии устранения дублирования)
- оставлять неиспользуемые импорты
- добавлять новые методы в MainWindow, которые можно вынести в отдельные классы

## 4. Критерии приёмки (ОТК) — расширенные

Работа считается принятой ТОЛЬКО ЕСЛИ ВСЕ ПУНКТЫ ВЫПОЛНЕНЫ:

### 4.1. Архитектурные критерии

- количество строк кода уменьшилось по сравнению с исходным состоянием
- исчезли дублирующие системы
- у формата нет собственного состояния
- ConversionEngine — единственный управляющий слой
- эффекты только декларативные
- FileCollector — один
- UI изолирован от домена
- тип формата выбран явно
- старый код удалён, а не закомментирован
- не создано новых файлов, дублирующих функциональность существующих
- все импорты расположены в начале файлов (не внутри методов)

### 4.2. Метрики рефакторинга

- Файл main_window.py не может содержать более 500 строк кода (в исходном состоянии было 2568). Любое превышение требует обоснования и согласования
- Ни один класс не может превышать 500 строк (исключение – классы, реализующие сложный интерфейс, с обоснованием)
- Ни один метод не может превышать 50 строк (кроме конструктора, инициализации UI)
- Процент дублирования кода (по инструментам типа pylint/radon) не должен превышать 5%

### 4.3. Чистота кода

- Нет закомментированного кода (кроме временных пометок, удаляемых до сдачи)
- Нет файлов с суффиксами _backup, _old, _deprecated в основном каталоге (они должны быть удалены или перенесены в legacy/)
- Нет дублирования классов (каждый класс определён ровно в одном месте)
- Все методы аннотированы типами (хотя бы базовыми)
- Нет print() в production-коде, только логирование через централизованный логгер

### 4.4. Тестирование (если применимо)

- Для изменённых модулей добавлены или обновлены unit-тесты, покрывающие новую функциональность
- Все тесты проходят успешно (проверка через pytest)

Работа отклоняется, даже если «всё работает», но нарушен хотя бы один из вышеперечисленных критериев.

## 5. Процесс рефакторинга

Любое изменение архитектуры должно следовать пошаговому плану:

1. **Анализ** – оценка текущего состояния, выявление дублирования, точек расширения
2. **Создание новых модулей** – вынос ответственности в отдельные классы (например, SettingsManager, HistoryManager)
3. **Поэтапное подключение** – замена вызовов в MainWindow на вызовы новых менеджеров, без поломки существующего функционала
4. **Удаление старого кода** – после того как новый код полностью покрывает функциональность, старый код удаляется (не комментируется, не оставляется в резерве)
5. **Проверка метрик** – убедиться, что количество строк уменьшилось, дублирование устранено
6. **Запуск тестов** – все тесты должны проходить
7. **Код-ревью** – изменения рассматриваются с точки зрения соблюдения контракта

## 6. Ответственность агента

Агент (AI или разработчик) несёт полную ответственность за соблюдение контракта.

Если после изменений выявляются нарушения (например, увеличение строк, дублирование, оставленный бэкап), работа считается невыполненной и подлежит переделке за счёт исполнителя.

Допустимые исключения – только с явным согласованием в комментариях к пулл-реквесту с указанием причины.

## 7. Статус контракта

Этот документ — единственный источник правды.

Любое отклонение требует явного согласования.

Архитектурная «гибкость» без согласования считается ошибкой.

## 8. Археология

**legacy/**:

- не изменяется
- не рефакторится
- используется только как справочник поведения

Копировать поведение — допустимо.
Копировать код — запрещено.

## Финальная ремарка (для агентов)

Архитектура — не про «чтобы всем было удобно».
Архитектура — это распределение власти, границ и ответственности.

Унификация ради унификации — форма технического самообмана.

Качественный рефакторинг — это не добавление новых файлов, а удаление лишнего и устранение дублирования. Если после ваших изменений строк кода стало больше — вы сделали что-то не так.
